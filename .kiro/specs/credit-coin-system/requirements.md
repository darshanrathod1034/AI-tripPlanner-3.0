# Requirements Document

## Introduction

The Credit Coin System is a gamification and referral feature for the Trip Planner application that manages user credits for AI-generated trip itineraries. The system provides default credits to new users, deducts credits when generating itineraries, and rewards users for successful referrals. This feature aims to incentivize user acquisition through referrals while controlling AI generation costs.

## Glossary

- **Credit_Coin_System**: The subsystem responsible for managing user credit balances, transactions, and referral codes
- **User**: A registered account holder in the Trip Planner application
- **Credit_Balance**: The number of credit coins available to a User
- **Credit_Transaction**: A record of credit coins earned or spent by a User
- **Referral_Code**: A unique alphanumeric identifier assigned to each User for referring new Users
- **Referring_User**: A User who shares their Referral_Code with potential new Users
- **Referred_User**: A new User who signs up using a Referral_Code
- **AI_Itinerary**: A trip itinerary generated by the AI service
- **User_Account**: The complete user profile including authentication and credit information
- **Frontend_Display**: The user interface component showing credit balance information
- **Signup_Process**: The user registration workflow including account creation and authentication

## Requirements

### Requirement 1: Default Credit Allocation

**User Story:** As a new user, I want to receive default credit coins when I create my account, so that I can immediately start generating trip itineraries without additional steps.

#### Acceptance Criteria

1. WHEN a User_Account is successfully created and persisted in the database, THE Credit_Coin_System SHALL create a Credit_Balance of 5 credit coins for that User
2. THE Credit_Coin_System SHALL create the Credit_Balance and record the Credit_Transaction as a single atomic operation during User_Account creation
3. WHEN the Credit_Balance is created, THE Credit_Coin_System SHALL record a Credit_Transaction with transaction type "signup_bonus" and amount 5
4. IF the User_Account creation fails or is rolled back, THEN THE Credit_Coin_System SHALL NOT create a Credit_Balance or Credit_Transaction for that User

### Requirement 2: Credit Deduction for AI Itinerary Generation

**User Story:** As a user, I want to spend credit coins to generate AI trip itineraries, so that the system can manage resource usage fairly.

#### Acceptance Criteria

1. WHEN a User requests an AI_Itinerary, THE Credit_Coin_System SHALL verify the User has a Credit_Balance greater than or equal to 1 before initiating generation
2. IF the Credit_Balance is less than 1, THEN THE Credit_Coin_System SHALL reject the AI_Itinerary request and return an error indication to the User
3. WHEN a User with sufficient Credit_Balance requests an AI_Itinerary, THE Credit_Coin_System SHALL deduct 1 credit coin from the Credit_Balance and record the Credit_Transaction as a single atomic operation
4. THE Credit_Coin_System SHALL complete the credit deduction before initiating the AI_Itinerary generation
5. THE Credit_Transaction SHALL include transaction type "itinerary_generation", amount -1, and timestamp
6. IF the AI_Itinerary generation fails within 60 seconds after credit deduction, THEN THE Credit_Coin_System SHALL refund 1 credit coin to the User
7. IF the AI_Itinerary generation does not complete within 60 seconds, THEN THE Credit_Coin_System SHALL refund 1 credit coin to the User and terminate the generation request
8. THE Credit_Coin_System SHALL record a Credit_Transaction with transaction type "refund" and amount 1 for each refund operation

### Requirement 3: Referral Code Generation

**User Story:** As a user, I want to have a unique referral code, so that I can invite friends and earn rewards.

#### Acceptance Criteria

1. WHEN a User_Account is created, THE Credit_Coin_System SHALL generate a unique Referral_Code for that User
2. THE Referral_Code SHALL contain exactly 8 characters from the set [A-Z, 0-9] excluding visually ambiguous characters [0, O, I, 1]
3. THE Credit_Coin_System SHALL verify the Referral_Code is unique across all Users before assignment
4. IF the generated Referral_Code already exists, THEN THE Credit_Coin_System SHALL generate a new Referral_Code and retry uniqueness verification up to 5 times
5. IF the Credit_Coin_System cannot generate a unique Referral_Code after 5 attempts, THEN THE Credit_Coin_System SHALL fail the User_Account creation with error indication "referral_code_generation_failed"
6. THE Referral_Code SHALL remain constant for the lifetime of the User_Account
7. THE Credit_Coin_System SHALL store the Referral_Code in the User_Account data before User_Account creation completes

### Requirement 4: Referral Code Validation

**User Story:** As a new user, I want to enter a referral code during signup, so that both the referring user and I can receive credit rewards.

#### Acceptance Criteria

1. WHERE a Referral_Code is provided during Signup_Process, THE Credit_Coin_System SHALL validate the Referral_Code exists in the system and return a validation result
2. IF the provided Referral_Code does not exist in the system, THEN THE Credit_Coin_System SHALL reject the Signup_Process and return an error indication to the User
3. IF the provided Referral_Code matches the User's own Referral_Code, THEN THE Credit_Coin_System SHALL reject the Signup_Process and return an error indication to the User
4. WHEN a valid Referral_Code is provided during Signup_Process, THE Credit_Coin_System SHALL grant 5 credit coins to the Referred_User and 25 credit coins to the Referring_User
5. IF the Referral_Code belongs to an inactive or deleted User_Account, THEN THE Credit_Coin_System SHALL reject the Signup_Process and return an error indication
6. THE Credit_Coin_System SHALL allow Signup_Process to proceed without a Referral_Code
7. WHEN Signup_Process proceeds without a Referral_Code, THE Credit_Coin_System SHALL grant 5 credit coins to the new User

### Requirement 5: Referral Reward Distribution

**User Story:** As a referring user, I want to receive credit coins when someone signs up with my referral code, so that I am incentivized to invite new users.

#### Acceptance Criteria

1. WHEN a Referred_User completes Signup_Process with a valid Referral_Code, THE Credit_Coin_System SHALL add 25 credit coins to the Referring_User Credit_Balance within 5 seconds
2. THE Credit_Coin_System SHALL add the referral reward and record the Credit_Transaction as a single atomic operation
3. THE Credit_Transaction for the Referring_User SHALL include transaction type "referral_reward", amount 25, and timestamp
4. THE Credit_Coin_System SHALL record the relationship between Referring_User and Referred_User in the User_Account data
5. THE Credit_Coin_System SHALL prevent duplicate referral rewards for the same Referred_User
6. IF the referral reward operation fails, THEN THE Credit_Coin_System SHALL roll back the Referred_User account creation
7. WHEN the Referred_User account creation is rolled back, THE Credit_Coin_System SHALL return an error indication to the Referred_User

### Requirement 6: Credit Balance Display

**User Story:** As a user, I want to see my current credit balance in the interface, so that I know how many itineraries I can generate.

#### Acceptance Criteria

1. WHEN a User is authenticated, THE Frontend_Display SHALL show the current Credit_Balance
2. THE Frontend_Display SHALL update the Credit_Balance within 2 seconds after a Credit_Transaction occurs
3. THE Frontend_Display SHALL show the Credit_Balance as a non-negative integer
4. THE Frontend_Display SHALL be visible on all pages where AI_Itinerary generation is available
5. THE Frontend_Display SHALL use a minimal and professional design that does not obstruct primary content

### Requirement 7: Credit Transaction History

**User Story:** As a user, I want to view my credit transaction history, so that I can track how I earned and spent credits.

#### Acceptance Criteria

1. THE Credit_Coin_System SHALL maintain a chronological record of all Credit_Transactions for each User
2. WHEN a User requests their transaction history, THE Credit_Coin_System SHALL return all Credit_Transactions sorted by timestamp in descending order
3. THE Credit_Transaction record SHALL include transaction type, amount, timestamp, and resulting Credit_Balance
4. THE Credit_Coin_System SHALL retain Credit_Transaction records for the lifetime of the User_Account

### Requirement 8: Referral Code Retrieval

**User Story:** As a user, I want to easily access my referral code, so that I can share it with potential new users.

#### Acceptance Criteria

1. WHEN a User requests their Referral_Code, THE Credit_Coin_System SHALL return the User's unique Referral_Code
2. THE Credit_Coin_System SHALL return the Referral_Code within 500 milliseconds
3. THE Frontend_Display SHALL provide a mechanism to copy the Referral_Code to the clipboard

### Requirement 9: Credit Balance Integrity

**User Story:** As a system administrator, I want credit balances to remain accurate and non-negative, so that the system maintains financial integrity.

#### Acceptance Criteria

1. THE Credit_Coin_System SHALL ensure Credit_Balance is never negative
2. WHEN a Credit_Transaction would result in a negative Credit_Balance, THE Credit_Coin_System SHALL reject the transaction with error code "insufficient_credits"
3. THE Credit_Coin_System SHALL use atomic database operations for all Credit_Balance modifications
4. THE Credit_Coin_System SHALL validate that the sum of all Credit_Transactions equals the current Credit_Balance

### Requirement 10: Concurrent Transaction Handling

**User Story:** As a user, I want my credit transactions to be processed correctly even when multiple requests occur simultaneously, so that my balance remains accurate.

#### Acceptance Criteria

1. WHEN multiple Credit_Transactions occur concurrently for the same User, THE Credit_Coin_System SHALL process them sequentially
2. THE Credit_Coin_System SHALL use database-level locking to prevent race conditions
3. THE Credit_Coin_System SHALL ensure each Credit_Transaction sees the correct Credit_Balance before modification
